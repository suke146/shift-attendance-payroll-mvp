# 残りの実装機能一覧

## 目的

現在のシフト管理アプリに対して、テスト前に追加したい機能を整理する。  
既存機能は、希望シフト提出、シフト制作、シフト管理、カレンダー表示、Excel/スプレッドシート出力、スタッフ管理、店舗管理、給料計算、給与ルール設定まで実装済み。

今後は、より実店舗運用に近づけるために以下の機能を追加する。

---

# 追加実装予定機能

## Step 23：シフト提出期限

### 概要

店舗ごと、対象月ごとに希望シフトの提出期限を設定できるようにする。

### 主な機能

- manager / admin が提出期限を設定
- staff は期限内のみ希望シフトを提出・削除可能
- 期限後は提出画面に「締切済み」と表示
- manager / admin は期限後も希望シフト一覧を確認可能
- 月ごとに締切日を変更可能

### 実装対象

- DB：`shift_submission_deadlines`
- 画面：`/admin/shift-deadlines`
- 影響画面：`/shifts/submit`

---

## Step 24：必要人数ルール・不足チェック

### 概要

曜日・時間帯・ポジションごとに必要人数を設定し、確定シフトと比較して不足人数を確認できるようにする。

### 主な機能

- 曜日別の必要人数設定
- 時間帯別の必要人数設定
- ポジション別の必要人数設定
- 日別の不足人数表示
- シフト制作画面で不足日を確認
- カレンダー上で不足日を強調表示

### 実装対象

- DB：`staffing_requirements`
- 画面：`/admin/staffing-requirements`
- 影響画面：
  - `/admin/shifts/create`
  - `/calendar`
  - `/dashboard`

---

## Step 25：不足募集機能

### 概要

必要人数に対してシフトが足りない日・時間帯に対して、manager がスタッフへ募集を出せるようにする。

### 主な機能

- manager が不足募集を作成
- staff が不足募集一覧を確認
- staff が勤務可能として応募
- manager が応募者を確認
- manager が応募を承認すると確定シフトに反映

### 実装対象

- DB：
  - `shift_openings`
  - `shift_opening_applications`
- 画面：
  - `/admin/shift-openings`
  - `/shift-openings`

# 注意点

## 実装時の注意

- 1機能ずつ実装する
- 各StepごとにGitHubへ保存する
- DB変更後はRLSを必ず確認する
- staff / manager / admin の権限差を毎回確認する
- 店舗IDによるデータ分離を守る
- 給与関連は「概算」として扱う
- 正式な給与計算・労務判断には使わない前提で表示する

## セキュリティ注意

- `.env.local` はGitHubに上げない
- Supabase service_role key はフロントエンドで使わない
- 給与情報は本人・manager・admin 以外に見せない
- 他店舗のデータが見えないようにRLSを確認する
- 操作履歴を残せるようにする

## 実務導入前に確認すること

- 店長が実際に使いやすいか
- スタッフがスマホで迷わず使えるか
- シフト提出から確定までの流れが自然か
- 不足募集や代打申請の承認フローが現場に合っているか
- 人件費計算が店舗運用の考え方と合っているか
- Excel / Googleスプレッドシート出力が実際の提出形式に合っているか
